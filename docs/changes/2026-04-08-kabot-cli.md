# 2026-04-08: kabot 간편 전송 CLI 추가

## 배경

기존 CLI는 `katalk-ax send --chat ... --message ...` 형태라 기능은 충분했지만, 사용자가 바로 입력하기에는 다소 길었습니다.

## 목표

`kabot --room "허동운" --message "안녕"` 같은 형태로 곧바로 전송 흐름에 들어갈 수 있는 더 단순한 CLI 진입점을 추가합니다.

## 변경 내용

- `Package.swift`에 `kabot` 실행 파일 타깃/제품 추가
- `Sources/KTalkAX/CLI/KabotCommandParser.swift` 추가
  - `--room` 지원
  - 사용자가 적은 `--rome` 별칭도 지원
  - 유니코드 em dash/엔대시를 `--`로 정규화
- `Sources/KabotCLI/main.swift` 추가
  - 공유 `KTalkAXService.send(...)`를 직접 사용
- `Tests/KTalkAXTests/KabotCommandParserTests.swift` 추가
- `README.md`, `scripts/quick-setup.sh`, `Formula/katalk-ax.rb`, `.github/workflows/release.yml`를 갱신해 배포/설치 경로에도 `kabot` 반영

## 설계 이유

- 기존 `katalk-ax` CLI를 그대로 유지하면서, 더 단순한 사용자 경험은 별도 실행 파일로 제공하는 편이 가장 안전합니다.
- `kabot`은 새 전송 로직을 만들지 않고 기존 공유 서비스 경로를 그대로 재사용합니다.
- 이렇게 하면 ambiguity 처리, 드라이런, 검증, KakaoTalk 휴리스틱이 모두 한 곳에 남습니다.

## 검증

실행으로 확인한 내용:

```bash
swift build
swift test
swift build --product kabot
swift run kabot --help
swift run kabot --room "허동운" --message "안녕" --dry-run --json
swift run kabot --rome "허동운" --message "안녕" --dry-run
```

관찰 결과:

- `kabot --help`는 한국어 도움말을 정상 출력
- `--room`과 `--rome` 둘 다 정상 파싱
- 현재 터미널 프로세스 접근성 권한이 없으면 기존 `katalk-ax send`와 동일하게 권한 오류로 중단
- 즉, 새 래퍼가 별도 구현이 아니라 기존 전송 경로를 제대로 타고 있음이 확인됨

## 영향

- 더 단순한 전송 중심 CLI 진입점이 생김
- 기존 `katalk-ax` 사용법은 그대로 유지됨
- Homebrew/release 아카이브에도 `kabot`이 포함됨

## 남아 있는 한계

- `kabot`도 결국 같은 접근성 권한을 필요로 하므로, 실행 주체에 권한이 없으면 전송 전 단계에서 중단됩니다.
